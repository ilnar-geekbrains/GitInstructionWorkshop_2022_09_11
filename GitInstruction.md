# Инструкция по работе с git
## Настройки
* Самые важные: наше имя пользователя и адрес электронной почты
```
    git config --global user.name "My Name"
    git config --global user.email myEmail@example.com
```
* Git хранит весь пакет конфигураций в файле .gitconfig, находящемся в вашем локальном каталоге. Чтобы сделать эти настройки глобальными, то есть применимыми ко всем проектам, необходимо добавить флаг –global.
* Для просмотра всех настроек системы:
```
    git config --list
```
* Для вывода общей документации (подсказок по всем командам):
```
    git --help
```
* Для вывода документации по конкретной команде прописываем в консоли:

```
    git log --help
```
## Создание нового репозитория
* Открываем папку с будущим проектом в консоли
* прописываем команду для инициализации git:
```
     git init
```
* после инициализации создаем рабочие файлы проекта
## Определение состояния
* status - команда, показывающая информацию о текущем состоянии репозитория
```
    git status
```
* при добавлении новых файлов при вызове команды git status выведется сообщение о том, что в системе появился неотслеживаемый файл (untracked files)
* для того, чтобы его начать отслеживать, необходимо его специальным образом объявить
## Подготовка файлов
* командой git add добавляем в область подготовленных файлов все внесенные изменения.
```
    git add hello.txt
```
* для добавления всего, что находится в директории, используем
```
    git add -A
```
* после данной команды файл готов к коммиту (созданию слепка нужного состояни, т.е. сохранению изменений под определенным хэшем)
* Для добавления всех файлов используется: 
```   
    git add .
```
* При добавлении ненужных элементов в область подготовленных файлов их можно убрать при помощи команды:
 ```
    git reset имя_файла.расширение
 ```
## Фиксация изменений
* Создание коммита:
``` 
    git commit -m "Add some code"
```
* Добавление файла в область подготовленных файлов и одновременное создание коммита выполняется при помощи команды:
```
    git commit -a -m "Add some code"
```
## Просмотр всех созданных коммитов
* Выполняется при помощи команды:
```
    git log
```
* Здесь хранится вся информация о каждом отдельном коммите, с указанием хэша, автора, списка изменений и даты, когда они были сделаны
* Отследить интересующие нас операции в списке изменений, можно по хэшу коммита, при помощи команды:
```
    git show hash_commit
```
* Если необходимо переделать commit message и внести туда новый комментарий, пишем следующую конструкцию:
```
    git commit --amend -m "New comment" 
```
* В случае описанном выше сообщение последнего коммита перезапишется (не рекомендуется злоупотреблять данной командой)
## Удаление локального репозитория
* Необходимо удалить скрытую папку .git в корне репозитория. Это можно сделать напрямую в катологе, через консоль и на Github
## Ветвление
* Создание новой ветки:
```
    git branch new_branch
```
* Посмотреть какие ветки есть
```
    git branch
```
* Переключение между ветками:
```
    git checkout new_branch
```
* Необходимо помнить, что пока мы не закоммитим собственные изменения переключиться между ветками не получится. 
* Для обхода данного запрета существует команда
```
    git stash 
```
* Она помещает текущие незакоммиченные изменения в стек изменений и сбросит рабочюю копию до HEAD репозитория
* Слияние веток:
    * В new_branch добавим изменения 
    `git add new_file.txt`
    `git commit -m "added new file"`
    * переключаемся обратно на ветку master
    `git checkout master`
    * для объединения веток воспользуемся `git merge new_branch`
    * если new_branch ветка больше не нужна, то ее можно удалить 
    `git branch -d new_branch`
    * если мы хотим создать копию удаленного репозитория, используем
    `git clone`
    * если мы хотим создать копию ветки удаленного репозитория, используем
    `git checkout -b имя_ветки origin имя ветки`
* Удаление веток
    * для локально расположенных веток
    `git branch -d local_branch_name`
    где -d сокращение --delete
    * но удалить ветку, если вы в ней находитесь нельзя, нужно перейти на основную, а потом произвести команду удаления

## Удаленные репозитории
* Работа с удаленным репозиторием
    * начинается с его клонирования к себе на компьютер
    `git clone https://github.com/my_project`
    Создастся папка с одноименным названием как на удаленном сервере
    * При необходимости клонировать папку со своим названием можно выполнить следующую команду:
`git clone https://github.com/my_project new-folder`
* Подключение к удаленному репозиторию
    * Чтобы загрузить что-нибудь в удаленный репозиторий, сначала нужно к нему подключиться.
    * Для связи локального репозитория с удаленным, выполним следующую команду:
    git remote add origin https://github.com/my_project.git
    * Обычно главный репозиторий называется origin
* Отправка изменений на сервер
    * Команда, предназначенная для перенесения локального коммита на сервер - push. Она принимает два параметра: имя удаленного репозитория (к примеру, origin) и ветку, в которую необходимо внести изменения (master - ветка по умолчанию для всех репозиториев)
    `git push origin master`
    * данная команда похожа на команду `git fetch`, с той разницей, что при помощи fetch мы импортируем коммиты в локальную ветку, а применив push, мы экспортируем их из локальной в удаленную
    * `git push` - команда выгрузки,
    * `git pull` - команда загрузки (скачивания),
    * `git fetch` - команда загрузки (скачивания),
    * после успешного перемещения на сервер измененных данных, их необходимо внедрить, при помощи команды слияния git merge
* Запрос изменений с сервера 
    * Если мы сделали изменения в своем удаленном репозитории, другие пользователи могут скачать изменения при помощи команды pull
    `git pull origin master`
## Как настроить совместную работу
1. Создать аккаунт на Github.com
2. Создать локальный репозиторий 
3. "Подружить" ваш локальный и удаленный репозитории
    Github при создании нового репозитория подскажет, как это сделать
4. Отправить (push) ваш локальный репозиторий в удаленный (на Github), при этом, возможно, вам нужно будет авторизоваться на удаленном репозитории
5. Произвести изменения "с другого компьютера"
6. Выкачать (pull) актуальное состояние из удаленного репозитория 

## Как сделать pull request
1. Делаем fork(ответвление) репозитория к себе
2. Делаем git clone СВОЕЙ версии репозитория
3. Создаем новую ветку и в НЕЕ вносим свои изменения
4. Фиксируем изменения (делаем коммиты)
5. Отправляем свою версию в свой Github
6. На сайте Github нажимаем кнопку pull request
## Дополнительно
* Отслеживание изменений, сделанных в коммитах
    * у каждого коммита есть свой уникальный идентификатор в виде цифр и букв
    * Идентификаторы длинные, но для работы с ними достаточно пяти первых символов
    * чтобы посмотреть, что нового появилось в коммите, можно воспользоваться командой 
    `git show b10cc123`
    * чтобы увидеть разницу между двумя коммитами, используется команда diff(с указанием промежутка между коммитами)
    `git diff 09bd8cc..ba25c0ff`
    * git difftool - запускает графический клиент, в которм наглядно сопоставляет все изменения
* Возвращение файла к предыдущему состоянию
    * Гит позволяет вернуть выбранный файл к состоянию на момент определенного коммита.
    `git checkout 09bd8cc1 hello.txt`
* Исправление коммита
    * commit —amend - эта команда добавит все из последнего коммита в область подготовленных файлов и попытается сделать новый коммит.
    * Для более сложных исправлений, например, не в последнем коммите или если вы успели отправить изменения на сервер, нужно использовать revert. Эта команда создаст коммит, отменяющий изменения, совершенные в коммите с заданным идентификатором.
    * Самый последний коммит может быть доступен по алиасу HEAD:
    `git revert HEAD`
    `git revert b10cc123`
* Настройка .gitignore
    * файл, с прописанными названиями файлов, которые не должны попасть в область подготовки файлов к коммиту
    * Создайте вручную файл под названием .gitignore и сохраните его в директорию проекта.
    * Внутри файла перечислите названия файлов/папок, которые нужно игнорировать, каждый с новой строки.
    * Файл .gitignore должен быть добавлен, закоммичен и отправлен на сервер, как любой другой файл в проекте.
    * Примеры файлов, которые нужно игнорировать:
        * Логи
        * Артефакты систем сборки
        * Папки node_modules в проектах node.js
        * Папки, созданные IDE, например, Netbeans или IntelliJ
        * Разнообразные заметки разработчика.
        * Файл .gitignore, исключающий все перечисленное выше, будет выглядеть так:
        ```
        *.log
        build/
        node_modules/
        .idea/
        my_notes.txt
        ```
